## 一、🧠 感知与理解层

### 1.1 多模态输入 — ❌ 严重缺失

| 项目 | 状态 |
|------|------|
| 文本输入 | ✅ |
| 图片输入 | ❌ |
| 音频输入 | ❌ |
| 视频输入 | ❌ |
| PDF/Office 文件解析 | ❌ |
| OCR | ❌ |
| 语音识别 | ❌ |

**根因分析**:

```python
# frontends/base.py — Message 模型只有文本
class Message:
    content: str  # 无 attachments / mime_type 字段
    role: str
    msg_type: MessageType
```

- `BaseFrontend` 的消息模型仅含 `content: str`，无 `attachments` 或 `mime_type` 字段
- `LLMClient` 无文件/图片 base64 编码发送能力
- `web_fetch` skill 可抓取网页内容，但仅限 URL，不是严格意义上的多模态输入

**改进方向** 🔧:
1. 扩展 `Message` 模型支持 `attachments: List[Attachment]`
2. `LLMClient` 支持 OpenAI Vision API (`image_url` content part)
3. 集成 `tesseract`/本地 OCR 管道处理图片文字
4. 添加 `speech_to_text` skill（whisper 本地模型）

### 1.2 多协议 LLM 支持 — ✅ 实现良好

`protocols/` 子包通过 `BaseProtocol` 抽象基类实现了三种协议适配：

| 协议 | 文件 | stream | tool_calls | 推理内容 |
|------|------|--------|------------|----------|
| OpenAI | `protocols/openai.py` | ✅ | ✅ | ✅ (DeepSeek R1/o1) |
| Anthropic | `protocols/anthropic.py` | ✅ | ✅ | — |
| Gemini | `protocols/gemini.py` | ✅ | ✅ | — |

**🔴 已修复 — Anthropic system prompt bug**:
原 `build_chat_request` 从 `kwargs['system']` 读 system prompt，但调用方将 system 放在 messages 列表中 (`role: "system"`)。Anthropic API 不允许 messages 中有 `system` role，导致 **HTTP 400 错误**。修复：`build_chat_request` 自动从 messages 列表中提取 system role 消息到专有 `system` 字段。 (`9f0310a`)

**🔴 已修复 — Gemini system/tool role bug**:
原 `build_chat_request` 将所有非 user 消息映射为 `model` role，system 消息未走 `systemInstruction` 字段，工具结果 (`role: "function"`) 也未正确处理。修复：system → `systemInstruction`，user/function → user role，tool → function role。 (`9f0310a`)

**小问题** ⚠️:

`BaseProtocol.parse_stream_chunk()` 签名返回 `Optional[str]`，但 Anthropic 的流式响应是 `ContentBlockStart/ContentBlockDelta` 结构（增量类型），目前通过 `_merge_tool_calls` 的通用处理兼容，但缺乏类型安全。

**改进方向** 🔧: `parse_stream_chunk` 返回更丰富的类型（如 `StreamChunk` dataclass）。

### 1.4 上下文理解 — ✅ 实现良好

`ChatCore` + `ContextCompressor` 三级压缩体系设计完善：

| 级别 | 触发条件 | 策略 |
|------|----------|------|
| NONE | token < 60% 阈值 | 不压缩 |
| MICRO | token < 80% 阈值 | 替换旧工具结果为占位符 |
| AUTO | token ≥ 80% 阈值 | 保存转录本 → 摘要 → 保留最近 K 轮 |
| MANUAL | 用户触发 | 全量摘要 + 仅保留最近 1 轮 |

**✅ 已修复 — 摘要质量** (`8dec199`):
`_generate_dialog_summary` 原为裸 `self.llm_client.chat(prompt)`，无 system_context/history/temperature 控制。修复：传入 `system_context="你是一个精准的对话摘要助手..."` + `max_tokens=512` + `temperature=0.3`。

**✅ 已修复 — 重复 token 计算** (`5a00b28`):
`process_context` 缓存命中分支中同一 messages 列表调用 3 次 `sum(count_tokens(...))`。修复：预计算 `total_original` 一次。

**🔴 已修复 — ContextCache 崩溃 bug** (`1c56d6a`):
`invalidate()` 和 `prune()` 使用 `sqlite3.connect(self.db_path)` 但 `ContextCache` 类没有 `db_path` 属性（实际应通过 `self.storage._get_connection()`），导致 **运行时 AttributeError 崩溃**。

**问题** ⚠️:
`auto_compact` 的实际触发路径——`ChatCore._compress_context` 返回压缩结果后，`chat_stream_with_tokens` 内部再次检查 token。两层检查不一致，可能导致 ChatCore 认为已压缩但实际仍超限。

## 二、🛠️ 行动与执行层

### 2.1 工具调用 (Function Calling) — ✅ 实现良好

`ToolRegistry` + `ToolExecutor` + 协议层 `build_chat_request_with_tools` 形成完整链条。

**架构**:
```
LLM response (tool_calls)
  → ToolExecutor.execute_tools_parallel()
    → ThreadPoolExecutor (max_workers=5)
      → ToolRegistry.execute_tool(name, arguments)
        → BaseTool.execute(**args)
```

**✅ 已修复 — 同步工具调用绕过 ToolExecutor** (`488f691`):
同步路径 (`_send_chat_request_with_tools`) 原直接调用 `execute_builtin_tool` → `ToolRegistry.execute_tool`，丢失了重试和超时控制。修复：统一使用 `self._tool_executor_instance.execute_single_tool`。

**问题 3** ⚠️: 敏感信息检测可绕过。`_redact_sensitive_info` 的正则 `rf'({pattern}[\s:=]+)\S+'` 假设敏感值无空格，含空格值时脱敏失败。
