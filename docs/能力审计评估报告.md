# Vermilion Bird 能力逐项审计评估报告

**审计日期**: 2026-05-03
**审计范围**: 全量源代码 (`src/llm_chat/`)，共 ~17,000 行
**审计方法**: 逐模块源码审查 + 架构分析

---

## 一、🧠 感知与理解层

### 1.1 多模态输入 — ❌ 严重缺失

| 项目 | 状态 |
|------|------|
| 文本输入 | ✅ |
| 图片输入 | ❌ |
| 音频输入 | ❌ |
| 视频输入 | ❌ |
| PDF/Office 文件解析 | ❌ |
| OCR | ❌ |
| 语音识别 | ❌ |

**根因分析**:

```python
# frontends/base.py — Message 模型只有文本
class Message:
    content: str  # 无 attachments / mime_type 字段
    role: str
    msg_type: MessageType
```

- `BaseFrontend` 的消息模型仅含 `content: str`，无 `attachments` 或 `mime_type` 字段
- `LLMClient` 无文件/图片 base64 编码发送能力
- `web_fetch` skill 可抓取网页内容，但仅限 URL，不是严格意义上的多模态输入

**改进方向**:
1. 扩展 `Message` 模型支持 `attachments: List[Attachment]`
2. `LLMClient` 支持 OpenAI Vision API (`image_url` content part)
3. 集成 `tesseract`/本地 OCR 管道处理图片文字
4. 添加 `speech_to_text` skill（whisper 本地模型）

---

### 1.2 多协议 LLM 支持 — ✅ 实现良好

`protocols/` 子包通过 `BaseProtocol` 抽象基类实现了三种协议适配：

| 协议 | 文件 | stream | tool_calls | 推理内容 |
|------|------|--------|------------|----------|
| OpenAI | `protocols/openai.py` | ✅ | ✅ | ✅ (DeepSeek R1/o1) |
| Anthropic | `protocols/anthropic.py` | ✅ | ✅ | — |
| Gemini | `protocols/gemini.py` | ✅ | ✅ | — |

**小问题**:

`BaseProtocol.parse_stream_chunk()` 签名返回 `Optional[str]`，但 Anthropic 的流式响应是 `ContentBlockStart/ContentBlockDelta` 结构（增量类型），目前通过 `_merge_tool_calls` 的通用处理兼容，但缺乏类型安全。

**改进方向**:
- 考虑 `parse_stream_chunk` 返回更丰富的类型（如 `StreamChunk` dataclass，含 `text`, `tool_call_delta`, `reasoning` 等字段）

---

### 1.3 意图识别 — ⚠️ 缺失

**现状**: 没有显式的意图识别层。LLM 的 function calling 隐式承担了意图路由。

**缺失能力**:
- 预分类器（快速路由到工具/问答/搜索，节省 token）
- "快速路径"（简单问候无需调用 LLM）
- 意图驱动的模型选择（简单问题用小模型，复杂推理用大模型）
- 用户指令前缀匹配（如 `/skill:name` 在 `prompt_skill.py` 中仅文档提及，未见运行时实现）

**改进方向**:
1. 添加 `IntentClassifier` 组件：正则/keyword + 轻量 embedding 分类
2. 实现 `/` 快捷指令路由（`/search`, `/file`, `/skill:name`, `/clear` 等）
3. 基于意图选择模型（config 中定义 `intent_model_map`）

---

### 1.4 上下文理解 — ✅ 实现良好

`ChatCore` + `ContextCompressor` 三级压缩体系设计完善：

| 级别 | 触发条件 | 策略 |
|------|----------|------|
| NONE | token < 60% 阈值 | 不压缩 |
| MICRO | token < 80% 阈值 | 替换旧工具结果为占位符 |
| AUTO | token ≥ 80% 阈值 | 保存转录本 → 摘要 → 保留最近 K 轮 |
| MANUAL | 用户触发 | 全量摘要 + 仅保留最近 1 轮 |

**问题 1**: `ContextCompressor._generate_dialog_summary` 调用 `self.llm_client.chat(prompt)`——

```python
# context/compressor.py:238
summary = self.llm_client.chat(prompt)  # 裸调用，无 system_context/history
```

无 `system_context`、无 `history`、无 tool awareness，摘要质量取决于裸 prompt。

**改进方向**: 使用 `generate()` API（`client/_generate.py`）并传入上下文。

**问题 2**: `auto_compact` 的实际触发路径不够 proactive。`ChatCore._compress_context` 返回压缩后的 `processed_history, processed_message`，然后发给 LLM。但 `chat_stream_with_tokens` 内部通过 `count_messages_tokens` 再做一次 token 检查——**两层检查不一致**，可能导致 ChatCore 认为已压缩但实际仍超限。

## 二、🛠️ 行动与执行层

### 2.1 工具调用 (Function Calling) — ✅ 实现良好

`ToolRegistry` + `ToolExecutor` + 协议层 `build_chat_request_with_tools` 形成完整链条。

**架构**:
```
LLM response (tool_calls)
  → ToolExecutor.execute_tools_parallel()
    → ThreadPoolExecutor (max_workers=5)
      → ToolRegistry.execute_tool(name, arguments)
        → BaseTool.execute(**args)
```

**问题**: 同步工具调用路径绕过了 `ToolExecutor`

```python
# client/_tools.py:_send_chat_request_with_tools
# 同步路径直接调用 execute_builtin_tool，走的是 ToolRegistry
# 而非 ToolExecutor — 丢失了重试和超时逻辑
tool_result = self.execute_builtin_tool(tool_call.name, tool_call.arguments)
```

`execute_builtin_tool` → `ToolRegistry.execute_tool` 直接执行，无重试、无超时控制。与流式路径 (`chat_stream_with_tools` → `ToolExecutor.execute_tools_parallel`) 行为不一致。

**改进方向**: 同步路径的统一使用 `self._tool_executor_instance.execute_single_tool`。

### 2.2 代码/Shell 执行 — ✅ 安全到位

| 安全机制 | 实现 |
|----------|------|

| 路径限制 | `os.path.abspath` + `startswith` 检查 |

**局限**: 白名单内的命令（如 `cat /etc/passwd`、`find / -name "secret"`）无法阻止，缺乏真正的容器化沙箱隔离。

### 2.5 任务委托与编排 — ✅⭐ 亮点

`task_delegator` skill 实现出色：

| 特性 | 实现 |
|------|------|
| 异步/同步模式 | `wait` 参数控制 |
| 工具白名单 | 父 agent 指定子 agent 可用工具 |
| 复杂度→模型映射 | `config.tools.subagent_models` (simple→小模型, complex→大模型) |
| 并发上限 | `subagent_max_concurrent` (默认 10) |
| 僵死检测 | `deadline` 超时自动标记 timeout |
| 重试 | 指数退避 (2, 4, 8 秒) |
| 资源清理 | `finally: client.close()` |
| 取消级联 | registry cancel callback → workflow cancel |

**工作流引擎** (`WorkflowExecutor`)：

| 模式 | 说明 |
|------|------|
| simple | 单个子 agent |
| parallel | 多个子 agent 并行（临时 ThreadPoolExecutor） |
| pipeline | 串行管道（前一输出→后一输入，`{parent_result}` 变量） |
| condition | 条件分支（基于前驱节点状态） |

**问题 1**: 子 agent 使用 `skip_skills_setup=True`

```python
# skills/task_delegator/tools.py:198
client = LLMClient(
    subagent_config, skip_skills_setup=True,
    tool_registry=self._tool_registry,
)
```

这意味着子 agent 不加载 skills，只依赖 `tool_registry` 中已注册的工具。好处是防止递归，坏处是**子 agent 无法使用任何 skill 的 `on_load` 配置逻辑**。例如 `shell_exec` 的白名单通过 `on_load` 注入，子 agent 中白名单为空或使用默认值。

**改进方向**: 为子 agent 保留可配置的 "skills subset"（只加载需要的 skills）。

**问题 2**: `GetSubagentStatusTool` 僵死检测的不完整覆盖

```python
# 如果 registry.submit 失败，_execute_async 从未被调用
# 则 context.deadline == 0，僵死检测被跳过
if context.deadline > 0 and time.time() > context.deadline:
```

**改进方向**: `spawn()` 时立即设置初始 deadline。

## 三、🧩 记忆与认知层

### 3.1-3.4 多层记忆系统 — ✅

四层记忆架构 + Markdown 文件持久化：

| 层级 | 文件 | 内容 | 生命周期 |
|------|------|------|----------|
| Short-term | `short_term.md` | 最近对话摘要 | 实时更新，最多 50 条 |
| Mid-term | `mid_term.md` | 每日摘要 + 事件时间线 | 保留 30 天，每 7 天压缩 |
| Long-term | `long_term.md` | 用户画像/重要事实/偏好 | 长期，每 7 天进化 |
| Soul | `soul.md` | 人格设定 | 用户可编辑 |

**问题 1**: 中期记忆无去重/合并

`compress_mid_term` 只删除超过 max_days 天的条目，但不会合并相似摘要。30 天内每天一条会导致文件膨胀。

**改进方向**: 使用 LLM 对近期摘要做主题聚类，合并相似条目。

**问题 2**: `build_system_prompt` token 预算优化

```python
# 只取最近 3 天，如果这 3 天摘要很短，预算被浪费
def _extract_recent_summary(self, content: str) -> str:
    for i in range(3):
        date = (today - timedelta(days=i)).strftime("%Y-%m-%d")
```

如果用户 3 天没使用，摘要为空，这部分 token 预算得不到利用。应该**动态分配**：先评估 high-priority 内容 token 数，剩余预算分配给 low-priority。

**问题 3**: 敏感信息检测可绕过

```python
SENSITIVE_PATTERNS = [r'密码', r'password', r'token', ...]
# 仅关键词匹配，'我的密 码是xxx' 可绕过
```

`_redact_sensitive_info` 的正则 `rf'({pattern}[\s:=]+)\S+'` 假设敏感词后紧跟 `\S+`（无空格的值），密码含空格时脱敏失败。

**改进方向**: 使用 LLM 辅助敏感信息检测 + PII 识别。

### 3.5 上下文压缩 — ✅

三级压缩 (MICRO/AUTO/MANUAL) + SQLite 缓存（`context_cache` 表）。

### 3.6 记忆进化 — ✅

`evolve_understanding()` 通过 LLM 检测并更新：语言偏好、回复风格偏好、代码风格偏好。每 7 天自动触发。

```python
module_name = f"skill_{skill_path.name}"
# 如果两个不同目录下有同名 skill，模块名冲突
```

**改进方向**: 使用完整路径 hash 作为模块名前缀。

### 4.2 MCP 协议 — ✅

`MCPManager` + `MCPClient` 支持 stdio/SSE 传输。

**问题 1**: `connect_all()` 同步阻塞 GUI

```python
# app.py:220
future = manager.connect_all()
results = future.result(timeout=180)  # 阻塞 3 分钟
```

GUI 启动时 `enable_tools()` 在主线程调用，3 分钟超时可能卡死 UI。

**改进方向**: 改为异步连接 + 进度回调，或在后台线程中执行。

**问题 2**: `MCPToolAdapter` 丢失 MCP 异步上下文

MCP 工具通过 `MCPToolAdapter(lambda name, args, mgr=manager: mgr.call_tool(name, args))` 包装，但 `call_tool` 是异步操作。在当前同步 `ToolRegistry.execute_tool` 路径中通过 `asyncio.run()` 桥接——如果有嵌套事件循环会报错。

### 4.3 多平台集成 — ✅

飞书集成：WebSocket 实时消息、Bot 入群事件、签名验证、多会话映射。

**问题 1**: 无自动重连

```python
# frontends/feishu/server.py:91
def _run_loop(self) -> None:
    self._client.start()  # 一次性启动，断开后不重连
```

飞书 WebSocket 断开后服务就死了，必须重启整个应用。

**改进方向**:

```python
def _run_loop(self) -> None:
    while not self._stop_event.is_set():
        try:
            self._client.start()
        except Exception:
            if not self._stop_event.is_set():
                time.sleep(self.reconnect_interval)
```

**问题 2**: `_handle_message_event` 无幂等/去重

飞书可能短时间重发同一事件，当前处理逻辑无去重机制，可能导致重复回复。

### 4.4 Webhook/API 暴露 — ❌ 完全缺失

无 HTTP API、无 Webhook 端点供外部系统调用。

### 4.5 数据库持久化 — ✅

SQLite WAL mode + 7 张表 + FTS5 + 原子迁移。

| 表 | 用途 |
|------|------|
| `conversations` | 会话元数据 |
| `messages` | 对话消息 |
| `messages_fts` | 全文搜索索引 |
| `tasks` | 定时任务定义 |
| `task_executions` | 任务执行记录 |
| `recent_feishu_chat` | 飞书会话记录 |
| `context_cache` | 上下文压缩缓存 |
| `apscheduler_jobs` | APScheduler 持久化 |

**问题**: 单连接模型（`contextmanager`），高并发子 agent 场景下可能成为瓶颈。无 PostgreSQL 支持（AGENTS.md 提到但未实现）。

## 五、🛡️ 安全与可靠性

### 5.1 本地优先 — ✅

所有数据存储在本地 SQLite + Markdown 文件。

**问题**: API Key 明文存储在 `config.yaml` / 环境变量。无 keyring 或加密存储。

`shell_exec` 有白名单 + 目录检查 + 超时，但**非真正容器化沙箱**。

**问题**: `_http_post_json_with_retry` 使用固定 1 秒 delay，应改为指数退避。

### 5.5 健康检查 — ✅

`HealthChecker` 注册/执行/汇总检查。已注册: database, services。

**缺失检查项**: LLM 连通性、MCP 服务器状态、磁盘空间、内存使用。
