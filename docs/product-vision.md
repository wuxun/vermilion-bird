# Vermilion Bird — 产品愿景与方向

> 面向未来的个人智能助手应该是什么样的
> 版本：2026-05-07 · 状态：方向文档

---

## 一、我们在做什么

Vermilion Bird 不只是"又一个 LLM 客户端"。我们正在构建一个**随着时间推移越来越"像你"的思考伙伴**——它记住你的过去、理解你的现在、预判你的需要，但永远把最终判断留给你。

现有架构（意图路由、四层记忆、沙箱执行、事件驱动）已经打下了很好的基础。本文档回答的是：**在此基础上，我们应该往哪里走，以及为什么。**

---

## 二、哲学根基

### 2.1 苏格拉底式对话，而非搜索引擎

最好的助手不是更快地回答你的问题，而是帮你**提出更好的问题**。

当一个用户说"帮我写个脚本"：

| 模式 | 行为 | 问题 |
|------|------|------|
| 搜索引擎模式 | 直接输出脚本 | 上下文缺失，可能错误 |
| **苏格拉底模式** | 先反刍：运行环境？安全边界？已有工具？ | 引导用户更清晰地定义问题 |

这意味着助手不应该把自己的能力包装成全知全能。它应该像优秀的工程师同事一样，**先理解再行动**。现有的 `IntentClassifier` 已经有意图路由的能力，但缺少"在行动前反刍确认"的环节。

### 2.2 渐进式亲密

信任不是靠显式问卷建立的，而是靠**长期对话的默默观察**。

助手应该自然而然地学习：

- 你习惯什么时候深度工作、什么时候摸鱼
- 你对代码质量的标准——是"跑通就行"还是"追求优雅"
- 你的决策偏好——冒险 vs 保守、抽象 vs 具体
- 你的沟通风格——简洁直接还是喜欢展开

这些不靠"请填写偏好设置"来获取。现有 `MemoryManager` 的四层记忆已经在积累信息，下一步是让这些记忆**驱动行为变化**，而不仅仅是存起来。

### 2.3 主权设计

> 你的助手应该忠于你，而非某个云端服务。

三个核心原则：

| 原则 | 含义 | 现有实现 | 差距 |
|------|------|----------|------|
| **本地优先** | 数据默认存在本地 | `Storage` (SQLite WAL + FTS5)、`MemoryStorage`（原子写入） | ✅ |
| **透明选择** | 发送到云端时告知用户 | — | 缺失：无隐私提示层 |
| **可迁移** | 换模型/厂商时，灵魂（记忆+偏好+风格）无缝跟随 | 记忆文件 `~/.vermilion-bird/memory/` 独立于模型 | ✅ |

关键缺口：**透明选择**。当前 pipeline（`ChatCore.send_message()` → `LLMClient` → 云端）没有在关键节点提示用户"这部分数据即将离开本地"。主权不只是一个技术属性，更是一种**可感知的设计**。

### 2.4 安静的能力

最好的工具是隐形的。理想助手应该在后台默默工作：

- 发现模式并提议自动化（"你每周五下午都做这个报告，要不要我定期生成草稿？"）
- 预处理信息流（摘要、分级、排队）
- **只在需要人类判断时才出现**

现有 `SchedulerService` + `WebhookServer` 已经有后台运行的能力，但行为是被动等待触发。真正安静的能力是**主动发现模式**——这是 `MemoryManager` + `Scheduler` 需要协同的地方。

## 三、用户体验金字塔

```
           ┌──────────┐
           │  L4 预见  │  ← 在你意识到之前，事情已经处理好了
          ┌┴──────────┴┐
          │  L3 主动   │  ← "这个会议和你的项目相关，要参加吗？"
         ┌┴────────────┴┐
         │  L2 对话深度  │  ← 记住上周聊过什么，能接上话
        ┌┴──────────────┴┐
        │  L1 即时响应   │  ← 问题→答案，快速准确
       ┌┴────────────────┴┐
       │  L0 无处不在     │  ← CLI/GUI/飞书/IDE，无缝切换
      └──────────────────┘
```

**Vermilion Bird 当前位置**：

| 层级 | 状态 | 说明 |
|------|------|------|
| L0 无处不在 | ✅ 良好 | CLI + GUI + 飞书三门面 |
| L1 即时响应 | ✅ 良好 | 多协议 LLMClient + 指数退避 + MCP 工具 |
| L2 对话深度 | 🔶 部分 | 四层记忆已有，但记忆驱动行为的能力不足 |
| L3 主动 | 🔶 基础 | scheduler + proactive 已架设，但缺少 pattern 发现 |
| L4 预见 | ❌ 未开始 | 需要 L2+L3 深度整合后才可能 |

向上走的路径不是堆更多功能，而是**让现有模块之间产生更深的共鸣**。

## 四、设计原则

### 原则 1：玻璃盒，而非黑盒

每当你看到助手做了一件事，应该能追问**"为什么？"**

| 用户追问 | 助手应能回答 | 相关模块 |
|----------|-------------|----------|
| "你为什么选 Claude 而不是 GPT？" | "长文档分析场景，Claude 上下文窗口更适合。" | `IntentClassifier` → 当前不暴露决策理由 |
| "你为什么提醒我这个？" | "过去 3 周你通常在周二处理这类任务，今天已是周三。" | `SchedulerService` → 无行为日志 |
| "你在做什么？" | "正在检索 7 天内的相关对话，已完成 3/5 项，预计剩余 2 秒。" | 缺失：执行透明度层 |

**当前差距**：`IntentClassifier` 有路由逻辑，`ToolExecutor` 有工具选择，但决策理由都没有暴露给用户。

**方向**：引入 `DecisionTrace` 机制——每次路由/工具调用/模型选择，生成一条可追溯的决策日志，在用户追问时即时呈现。

### 原则 2：光谱式自主权，而非开关

不是"自动执行"和"手动确认"的二选一，而是一个**连续的光谱**：

| 级别 | 命名 | 行为 | 适用场景 |
|------|------|------|----------|
| L0 | 观察 | 只回答，从不行动 | 敏感财务操作、安全相关 |
| L1 | 建议 | 提出行动方案，等你决策 | 策略性决策、重要修改 |
| L2 | 草拟 | 生成草稿，等你审批 | 对外沟通、代码提交 |
| L3 | 通知 | 执行后通知结果 | 常规操作、信息采集 |
| L4 | 自主 | 自主执行（有明确边界） | 定时备份、数据同步 |

**关键设计**：

1. **不同任务不同级别**：代码提交 = L2，信息搜索 = L4
2. **可临时覆盖**：即使默认 L4 的任务，也可以临时降为 L1
3. **边界声明**：每个自主行为都有声明过的边界，越界自动降级为 L1

**现有基础**：`SandboxExecutor` 已有边界控制（Docker/bwrap 隔离），`SchedulerService` 已有自动执行。缺少的是这个"信任度拨盘"的显式建模。

**配置化方向**：

```yaml
autonomy:
  default_level: L2          # 默认：草拟→审批
  per_skill:
    shell_exec: L1           # Shell 操作：建议
    web_search: L4           # 搜索：自主
    file_write: L2           # 文件写入：草拟→审批
  escalation:
    # 超出预算 → 自动降级到 L1
    max_cost_per_action_usd: 0.50
    max_concurrent_actions: 3
```

### 原则 3：人格不是"人设"，而是"镜面"

理想助手不应该有固定人设（像某些产品故意设计的俏皮助手），而应该是**你的镜面**——反映你的沟通风格、工作节奏、思考深度。

你简洁它就简洁，你爱深入它就陪你深入。

**现有基础**：`memory/templates.py` 有 6 种风格预设（default/academic/casual/concise/coach/architect），但需要手动切换。

**方向**：从手动切换 → **自动适应**。

| 信号 | 行为调整 |
|------|----------|
| 用户消息平均长度 | 调整回复详略程度 |
| 是否使用 `/` 快捷指令 | 调整交互模式偏好（指令型 vs 对话型） |
| 对建议的采纳率 | 调整主动建议的频次和置信度阈值 |
| 时间模式 | 工作时间精确模式，晚间简洁模式 |

这些信号已经存在于 `Storage` 的对话历史中，只是没有被分析和应用。

### 原则 4：记忆即身份

> 助手对你的记忆 = 你们共同的身份。

现有四层记忆：

| 层级 | 作用 | 现有状态 | 需要演化 |
|------|------|----------|---------|
| **short** | 本次对话上下文 | ✅ | — |
| **mid** | 跨对话主题聚类 | ✅ | — |
| **long** | 长期沉淀的知识 | ✅ | — |
| **soul** | 最核心的身份定义 | 🔶 静态模板 | **→ 动态演化** |

**Soul 应该是动态的**：

- 每次重要对话后，助手应该默默自问："我对这个人的理解有更新吗？"
- 不只是存储"用户说过什么"，还存储**"我们的关系如何"**——信任度、默契度、甚至你什么时候对助手失望过
- Soul 的演化应该是**缓慢且可追溯的**："我在 2026-05-01 更新了对你的理解：你更偏好 Rust 而非 C++，因为……"

## 五、三个最重要的下一步

### Next 1：从反应到预见

> 让 scheduler + memory + intent 协同，发现模式并主动提议，而非等待指令。

**具体场景**：

```
[周五 15:30]
助手: 我注意到过去 3 周你每周五下午都做了一份周报。
     需要我基于本周的对话摘要生成一个草稿吗？
     [生成草稿] [不需要] [了解更多]
```

**实现路径**：

1. `SchedulerService` 新增 `PatternDetector`：定期分析对话历史，发现重复行为模式
2. `MemoryManager` 提供行为模式查询 API
3. `ProactiveAgent` 将发现包装为决策卡片（`DecisionCard`）
4. 推送至用户选择的前端（GUI 通知 / 飞书消息）

**关联已有模块**：`scheduler/` + `memory/manager.py` + `decision/` (已在 plan 中)

### Next 2：透明决策链

> 让路由选择、工具调用、记忆提取都有可追问的"why"。

**具体场景**：

```
用户: 你为什么用了 Claude？
助手: （展开决策链）
      → 意图分类: 你问的是文档分析类问题
      → 路由规则: 长文档分析 → Claude Sonnet（上下文窗口 200K，优于 GPT-4o 的 128K）
      → 置信度: 高 (92%)
      → 备选: GPT-4o（如果 Claude 不可用）
```

**实现路径**：

1. 在 `IntentClassifier.classify()` 产生 `RoutingDecision` 时，同时产出 `DecisionTrace`
2. `ToolExecutor.execute_tools_parallel()` 记录工具选择理由
3. `MemoryManager` 的记忆提取/去重操作附带决策元数据
4. GUI 新增 `DecisionTraceView` 组件：可折叠的决策链展示

**关联已有模块**：`intent/classifier.py` + `tools/executor.py` + `memory/manager.py`

### Next 3：动态 Soul

> 让 soul 层从静态模板变成持续演化的、真正反映使用者本质的东西。

**具体场景**：

```
# Soul 演化日志
version: 27
last_evolved: 2026-05-07
changes:
  - reason: "你拒绝了 3 次 Rust 重构建议，更倾向于渐进式改进"
    confidence: 85
    updated:
      preferences.engineering_style: "稳步重构" (was "激进重写")
  - reason: "你在所有回答中都使用中文，即使在英文技术问题后"
    confidence: 97
    updated:
      preferences.language: "zh-CN 优先，即使面对英文输入"
willing_to_share: true
```

**实现路径**：

1. `MemorySoul` 从纯文本模板变为结构化（YAML/JSON）的动态对象
2. 每次重要对话后，`MemoryManager` 触发 `SoulEvolution` 检查
3. 只有当信心足够高（>75%）时，才更新 soul
4. Soul 版本化，每次更新可追溯
5. 新的 Soul 信息自动注入 system prompt，影响后续对话行为

**关联已有模块**：`memory/manager.py` + `memory/templates.py`

## 六、与现有模块的映射

| 愿景要素 | 关联模块 | 差距 |
|----------|----------|------|
| 苏格拉底式反刍 | `IntentClassifier` | 缺少"行动前确认"环节 |
| 渐进式亲密 | `MemoryManager` | 记忆已有，驱动行为不足 |
| 主权·透明选择 | `ChatCore` | 无隐私提示层 |
| 安静的能力 | `SchedulerService` + `MemoryManager` | 缺少 PatternDetector |
| 玻璃盒 | `IntentClassifier` + `ToolExecutor` | 无 DecisionTrace |
| 光谱自主权 | `ToolExecutor` + `SandboxExecutor` | 无 AutonomyLevel 建模 |
| 镜面人格 | `memory/templates.py` | 从手动到自动适应 |
| 动态 Soul | `memory/manager.py` | 从静态模板到演化对象 |
| 预见 | `scheduler/` + `proactive/` | 从反应到主动发现 |

## 七、不做的事情

为了保持清晰，以下方向明确不做：

1. **不做聊天机器人的人格扮演**：助手不假装是人，不在回答中表达情感。镜面原则意味着 reflecting 而非 role-playing。
2. **不做社交媒体式的"智能体广场"**：一个用户一个助手，关系深度优先于功能广度。
3. **不做封闭生态**：不锁定特定模型、特定云服务。可迁移性是一等公民。
4. **不做 AGI**：这仍然是一个工具，不是一个"有自我意识的存在"。

## 八、一句话

> 理想的未来助手不是一个更聪明的 ChatGPT，
> 而是一个**随着时间推移越来越"像你"的思考伙伴**——
> 它记住你的过去、理解你的现在、预判你的需要，
> 但永远把最终判断留给你。
